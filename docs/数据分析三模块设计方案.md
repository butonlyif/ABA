# 数据分析三模块设计方案

> 范围：进展记录 / 数据看板 / 报告中心
> 目标：让数据反映「时间趋势」而非当下快照，让报告从「总结」升级为「下一阶段处方」
> 状态：✅ 已实现（2026-06-10，见 `src/MVP_web/app_prototype.py`）

---

## 0. 现状诊断

| 模块 | 当前实现 | 问题 |
|---|---|---|
| 📊 进展记录 | 按技能折线、训练时间线、备注汇总 | 与数据看板**重叠**（都画每技能正确率） |
| 📈 数据看板 | 5 个指标卡 + 近30天频次 + 各技能最新% + 近10次辅助趋势 | 指标为「总量/最新」，**看不出快慢与阶段对比** |
| 📝 报告中心 | 训练数据拼成文字摘要，存于 tasks 表 | **只总结不导向行动**；借用任务表存储不理想 |

两个核心缺口：
1. **时间维度只到「最近一次」** —— 看不出辅助是否在退、掌握快不快、本期 vs 上期。
2. **报告止于总结** —— 没有把数据转成下一阶段能直接执行的动作。

数据基础（已具备，无需改结构）：
- 每次试次记录 `I/V/M/P/E`（独立正确 / 语言提示 / 示范 / 身体辅助 / 错误）
- `mastery_status()`：连续 3 次 ≥80% 独立 = 掌握，返回 `streak`
- 课程分步序列：初级 / 中级 / 高级技能分步训练（掌握后可指向下一步）

---

## 1. 最重要的数据（ABA 视角，按重要性）

现在只用了「独立正确率」一个数，但 `I/V/M/P/E` 是金矿。真正决定干预成败的 5 个信号：

1. **独立度趋势（去辅助曲线）** —— 不只看正确率高低，要看**辅助在不在减少**。
   80% 但全靠身体辅助，和 70% 全独立，意义相反。**ABA 最核心的信号。**
2. **掌握速度 / 停滞预警** —— 练了 6+ 次仍原地（独立度持平或下降）= 方案需要改。
   **这是报告能给出行动建议的关键触发器。**
3. **累计掌握技能数（随时间增长）** —— 家长最需要的「在进步」的大图景。
4. **维持情况** —— 掌握过的技能是否退步（需定期回测）。
5. **训练剂量 / 一致性** —— 练得太少是进展慢最常见的原因，要一眼可见。

---

## 2. 把「时间」放进去

把每个指标从「一个数」升级为「一条随时间的线 / 一个对比」：

- **去辅助堆叠面积图**：横轴时间，纵轴 100%，分层 I/V/M/P/E 占比。
  健康方案 = 绿色（独立）层随时间从下往上长满。一张图说清「在不在退辅助」。
- **累计掌握曲线**：横轴时间，纵轴累计掌握技能数，向上的阶梯 —— 给家长的成长故事。
- **阶段对比（本期 vs 上期）**：每个指标带 `↑+3` 的 delta（`st.metric` 原生支持，零成本）。
- **停滞标记**：自动算「连续 N 次没进步」，在进展记录里打 🟥 标签。

---

## 3. 三个模块重新分工（消除重叠）

- **📈 数据看板 = 此刻的体检报告**（全局 / 横向）
  累计掌握曲线 + 本期 vs 上期 delta + 剂量热力图 + 停滞技能红榜。回答「整体怎么样」。
- **📊 进展记录 = 单个技能的病历**（纵向 / 深入）
  选一个技能，看去辅助堆叠图 + 每次试次序列 + 备注。回答「这一项怎么练的」。
- **📝 报告中心 = 阶段总结 + 下一步处方**（见第 4 节）。

---

## 4. 报告 = 处方，而非总结（核心）

在报告中新增 **「下阶段建议」** 段，用规则从数据自动推导，每条技能给一个动作：

| 数据状态 | 判定条件 | 自动建议（可执行） |
|---|---|---|
| ✅ 已掌握 | 连续 3 次 ≥80% 独立 | 转入**维持期**（每周回测 1 次）+ 推进课程**下一分步** |
| 🟥 停滞 | 6+ 次仍 <80% 且持平/下降 | 改方案：拆更小步骤 / 加大辅助再退 / 换强化物 |
| 📈 进步中 | 独立度上升趋势 | 保持，下周加量 |
| ⚠️ 假掌握 | 正确率高但主要靠辅助（V/M/P 占比高） | 进入**退辅助**流程，暂不判掌握 |

闭环：**训练 → 数据 → 报告 → 新任务 → 训练**。
掌握后自动指向课程分步序列的下一步，报告直接接上下一阶段实践。

---

## 5. 页面要写清楚「这些数据怎么被用」

不能只摆图表，每块数据旁边要有一句话告诉家长「看到这个该做什么」。在各模块加引导文案（`st.caption` / `st.info` / 图表标题下一行小字）：

**📈 数据看板**
- 累计掌握曲线：「这条线在往上走 = 孩子在稳步进步；走平了说明近期没有新技能达标，看看是不是练得太少或卡在某项。」
- 去辅助趋势：「绿色（独立）越占越多 = 在成功退辅助；如果一直靠黄/橙/蓝（提示/示范/身体辅助），说明还不能算掌握。」
- 本期 vs 上期 delta：「绿色 ↑ 是比上周进步，红色 ↓ 要留意，结合备注找原因。」
- 剂量/频次：「ABA 讲究高频短时。本周次数偏低时，优先补训练量，再谈方法。」

**📊 进展记录**
- 🟥 停滞标签：「这项练了多次没进步，别硬练 —— 翻到报告看『下阶段建议』，多半要拆步骤或换强化物。」
- 试次序列（🟢🟡🟠🔵🔴）：「一行看懂这次哪几次独立、哪几次还要帮。红点多 = 任务可能太难。」

**📝 报告中心**
- 报告顶部固定一句：「本报告不只是总结 —— 往下看『下阶段建议』，每项技能都给了下周具体该怎么做。」
- 每条建议旁标注依据：「✅已掌握→维持+下一步 / 🟥停滞→改方案 / 📈进步中→加量 / ⚠️假掌握→先退辅助」，让家长知道建议从哪来、可信。

原则：**文案讲「怎么用」，不讲「这是什么」**。家长要的是下一步动作，不是统计学定义。

## 6. 报告每周自动生成

**实现方式：打开即补（纯前端，推荐）**
- 应用加载时检查该孩子最近一份周报的生成日期；若距今 ≥7 天（或已跨自然周），自动生成上一周的周报并存入历史。
- 在报告中心顶部用提示条告知：「📬 已为你生成上周报告」，避免家长以为要手动点。
- 优点：零额外部署，复用现有 `_build_report_text` + 报告存储。
- 局限：用户一周不打开就不会生成（对家长工具可接受，报告本就给打开应用的人看）。

**备选：真正定时（需外部调度）**
- 系统 cron / 部署平台定时任务跑独立脚本，遍历用户生成周报写库，用户不开也有。
- 代价：多维护一个脚本和写入路径，需处理「无登录态下访问数据」。仅在确有「不开也要推送」需求（如配合邮件/微信通知）时再做。

**本方案采用「打开即补」**；如需主动推送再升级为定时。
